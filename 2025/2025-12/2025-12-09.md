# **❗️なぜ先に削除しちゃダメなの？**
GraphQL のコード（schema / resolver / type 等）をフロントの対応より先に削除すると：
✔ フロントがまだその role を使っている

✔ フロントのコードが schema に問い合わせ続けている
この状態だと **フロントが死ぬ（クエリエラーになる）**
だから GraphQL 側の削除は **最後の方** に回す必要がある。


# **🔍 1. schema.graphql**
📌 **GraphQL サーバーの仕様そのものを書いたファイル**
特徴：
✔ backend エンジニアが手書きすることが多い
✔ schema-first 開発の中心
✔ API の仕様書の役割も持つ
↔ API の「憲法」みたいな存在
🔍 2. graphql.schema.json
📌 上の schema を JSON 形式に変換した “機械向け Schema ファイル”
特徴：
✔ 人間はほとんど直接見ない
✔ codegen（型生成ツール）が読み込む
✔ ESLint/IDE/型チェックのための metadata になる
↔ schema.graphql の機械翻訳版
🔍 3. graphql.ts
📌 クライアントが GraphQL を使うための TypeScript 型 & helper
主に codegen で生成されるファイル。
特徴：
✔ Frontend が import して使う
✔ クエリの型チェックや autocompletionが効く
✔ Apollo / React Query / urql などが利用する
↔ “schema.graphql から自動生成された TypeScript の API 契約書”
