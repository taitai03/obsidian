# RESTful APIとは
RESTの原則に則って構築されたwebシステムのHTTPの呼び出しインターフェースのこと

# RESTの原則
- <span style="background:#fff88f">アドレス可能性(Addressability)</span>:提供する情報がURIを通して表現できること。すべての情報はURIで表現される位置いいなアドレスを持っていること
- <span style="background:#fff88f">ステートレス性(Stateless):</span>HTTPをべおすにしたステートレスなクライアント/サーバプロトコルであること。セッション等の状態管理はせず、やり取りされる情報は知れ自体で完結して解釈できること。
- <span style="background:#fff88f">接続性(Connectability):</span>情報の内部に、別の情報や状態へのリンクを含めることができること
- <span style="background:#fff88f">統一インターフェース(Uniform Interface):</span>情報の操作はすべてHTTPメソッドを利用すること

# eager_load preloadの選択基準
[[2026-01-24|以前まとめたとこ]]

前提
preload
- 複数クエリ
- JOINしない
- associationを後からまとめて取得

eager_load
- 一つの巨大なJOINクエリ
- association構造を正確にSQLに反映
- JOINできないものは即エラー（モデルの理解が必要）


思考の軸①：壊れやすさを許容できるか
**preload を選ぶべき思考**
- 将来 association が変わるかもしれない
- メソッドの中身まで追いきれない    
- 「まず動く」が最優先
 **eager_load を選ぶべき思考**
- 仕様が固まっている
- association 構造を把握している
- 壊れるなら **早く壊れてほしい**


思考の軸②：「メソッド越しの association」があるか

フローチャート
```
strict_loading を使っている？
 ├─ NO → preload
 └─ YES
     ├─ association 構造を完全に把握している？
     │    ├─ NO → preload
     │    └─ YES
     │         ├─ バッチ / Mailer / Job？
     │         │    ├─ YES → eager_load
     │         │    └─ NO → preload
```

# ハッシュ、シンボルについて理解する

### シンボルと文字列の違い
シンボルは任意の文字列と一対一に対応するオブジェクト
<span style="background:#fff88f">同じ内容のシンボルは必ず同一オブジェクト</span>

`: シンボルの名前`
大量の文字列と大量のシンボルを使った場合、後者の方がメモリの使用効率もいい
シンボルを使ってハッシュを表現する
第一段階
currencies　=　{:japan => 'yen'}
値を取り出すとき
`currencies[:us]
シンボルがキーになる場合、=>を省略して書くことができる

currencies　=　{japan: 'yen'}
値を取り出すとき
`currencies[:us]
<span style="background:#fff88f">撮り方は同じことに注意</span>

値もシンボルだと以下のようになる
currencies　=　{japan:  :yen}


`currencies[:us]

### RSpec.describe
- これから各テストの対象を宣言するメソッド
- 第一引数にテスト対象の名前を指定
- type: :taskは、このテストがrakeタスクようであることをRspecに知らせるためのオプション

### let/let!
- Rspecで値を定義するためのメソッド
- letは呼ばれた時に初めて評価される
- let!はテスト実行時に必ず評価される

ex)
``` ruby
let(:user) { create(:user) } # 呼ばれるまで DB に作られない
let!(:user) { create(:user) } # テスト開始時点で DB に作られる
```

### before ... end ブロック
- 各it ブロックの前に必ず実行される処理を定義

### Rake.application.rake_require('...')
- Rakeタスクを読み込むメソッド

### Rake::Task.define_task(:environment)

- rakeタスクは通常 :environment に依存する
- テスト用に空の環境タスクを定義して依存関係を満たす。


### describe / context / it の違い
describe:テスト対象の「クラブ」や「メソッド」などを書く
context:テストしようとしている状況を書く
it:個別のテストの期待する結果を書く
1. テストの対象に対して
2. 特定の条件において
3. 期待するアウトプットが返ってくるかを調べる
この流れに沿うようにそれぞれ用いる

マッチャー一覧
https://zenn.dev/yuta0824/articles/ec3b985ffa36f0

マッチャーの選び方
	•	オブジェクトや値の変化をテストしたいとき
	•	.to change(Model, :count).by(x) → DBにレコードが増えたか
	•	値や配列に含まれるかを確認したいとき
	•	.to include(value)
	•	条件を満たす型や値か
	•	.to be_a(Class)
	•	.to be_kind_of(Class)
	•	.to eq(value)

💡 基準：
	•	副作用（DB, ファイル, メール送信）を検証したい → change, have_enqueued_job
	•	単純に値の比較 → eq, include, match

メールや外部呼び出しのテスト

例：Mailerのメソッドチェーン
```
expect(CompanyMailer).to receive(:with)
  .with(message: kind_of(CustomerMessage), entry_id: entry.id, user: entry.user)
  .and_return(double(notify_customer_message_received: double(deliver_later: true)))
```

	1.	Mailer呼び出しをテストしたいが、実際にはメールを送らない
	2.	.and_return(double(...)) → 戻り値をダミーに置き換え
	3.	ダミー内でさらに .notify_customer_message_received や .deliver_later を定義
	4.	メソッドチェーン全体がテスト可能になる
