# includes eager_load preload について

preload:指定したassociationを複数のクエリに分けて引いてキャッシュする。繰り替えし処理を実行するたびに、テーブルに問い合わせる必要がなくなる。つまり、大量のクエリを発行する必要がなくなり、N+１問題が解決される

eager_load:左外部結合(LEFT OUTER JOIN)でキャッシュする。また、特徴としてassociationの値で絞り込みをすることができる。

左外部結合とは：二つのテーブルの特定のフィールド同士を対応づけて、両者の値が一致するレコードを一つに連結して取得するもの。この時、一致する相手がいなかったレコードをデータとして取得するのが特徴

preloadを推奨するケース
- has_manyの関連を持つデータの事前読み込みを場合
- 複数の関連先の事前読み込みを行う場合

eager_loadでは常に結合処理を行うため、データ量が大きいと考えられる条件下では、重いクエリを実行することになってしまう。よって、クエリを分割して事前読み込みを行った方がレスポンスが早くなると考えられる。

eager_loadを推奨するケース
- 関連先の要素を絞り込みを行いたい場合
- has_one,belongs_toなど１クエリで取得した方が効率が良いと考えられる場合

関連先が上記のように１：１やN：１で関連づけられている場合、外部結合しても取得するレコードの数に変わりなく、１クエリでデータだ取得できるためpreloadより効率がいいと考えられます



# AplicationMailer

```ruby
class ApplicationMailer < ActionMailer::Base
  default from: "from@example.com"
  layout 'mailer'
end
```

ジェネレータを使用しない場合は、app/mailersディレクトリ以下にファイルを作成し、`ActionMailer::Base`を継承する


メーラーはrailsのコントローラーと非常に似ている。メーラーには「アクション」と呼ばれるメソッドがあり、メールのコンテンツを構成するのにビューを使用する。コントローラでHTMLなどのメールコンテンツを生成して顧客に送信したい場合、その箇所でメーラーを使用して、送信したいメッセージを作成します。


```ruby
class UserMailer < ApplicationMailer
  default from: 'notifications@example.com'

  def welcome_email(user)
    @user = user
    @url  = 'http://example.com/login'
    mail(to: @user.email, subject: 'Welcome to My Awesome Site')
  end
end
```
