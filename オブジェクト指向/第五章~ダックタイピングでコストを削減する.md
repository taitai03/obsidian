# 5.1ダックタイピングを理解する
型：変数の中身の分類

変数の中身の分類の知識、すなわち型についての知識があればア<span style="background:#fff88f">プリケーションの中身の振る舞いを予想することができる</span>
←Rubyにおける、オブジェクトの振る舞いについての一連の想定は、<u>パブリックインターフェースへの信頼</u>という形で行われる


しかしながら、クラスという方法で獲得するパブリックインターフェースは、オブジェクトが持ついくつかのパブリックインターフェースのうちの一つでしかないこともある

ここで大事なことは、<span style="background:#fff88f"><font color="#ff0000">オブジェクトが何で「ある」かではなく、何を「する」か</font></span>

柔軟な設計を作ることもできれば、理解不能なほど混沌とした、恐ろしい設計を構築することもできるからこそ、パブリックインターフェースは<span style="background:#fff88f">意図を持って入念に作る必要がある</span>

### 問題を悪化させる

```ruby

class Trip
  attr_reader :bicycles, :customers, :vehicle

  def prepare(preparers)
    preparers.each do |preparer|
      case preparer
      when Mechanic
        preparer.prepare_bicycles(bicycles)
      when TripCoordinator
        preparer.buy_food(customers)
      when Driver
        preparer.gas_up(vehicle)
        preparer.fill_water_tank(vehicle)
      end
    end
  end
end

# TripCoordinator と Driver を追加した
class TripCoordinator
  def buy_food(customers)
    # ...
  end
end

class Driver
  def gas_up(vehicle)
    # ...
  end

  def fill_water_tank(vehicle)
    # ...
  end
end
```
このコードは自身を出口のない苦境に陥れるプロセスの第一歩


引数はそれぞれ異なるクラスのもので、異なるメソッドを実装している

メソッドは他のクラスに依存していて、他のクラスは役に立たない。さらに、これらの具体定なメソッド名に依存している

このようなコードが蓄積されていくと、硬直して柔軟なアプリケーションを作ることができない

### ダックを見つける

依存を取り除くための鍵：<span style="background:#fff88f">「Tripのprepareメソッドは単一の目的を果たすためにあるので、その引数も単一の目的を共に達成するために渡されてくること」</span>

prepareメソッドにおいて、引数のクラスは想定されていない。かわりに、それぞれが「準備するもの(Prepare)」であることが想定されている
![[スクリーンショット 2026-02-02 13.14.12.png]]

![[スクリーンショット 2026-02-02 13.19.47.png]]

prepare_tripを実装するオブジェクトは、Preparerです

```ruby
# 旅行の準備はよりかんたんになる
class Trip
  attr_reader :bicycles, :customers, :vehicle

  def prepare(preparers)
    preparers.each do |preparer|
      preparer.prepare_trip(self)
    end
  end
end

# すべての準備者（Preparer）は
# 'prepare_trip' に応答するダック
class Mechanic
  def prepare_trip(trip)
    trip.bicycles.each do |bicycle|
      prepare_bicycle(bicycle)
    end
  end

  # ...
end

class TripCoordinator
  def prepare_trip(trip)
    buy_food(trip.customers)
  end

  # ...
end

class Driver
  def prepare_trip(trip)
    vehicle = trip.vehicle
    gas_up(vehicle)
    fill_water_tank(vehicle)
  end

  # ...
end
```
prepareメソッドは、新しいPreparerを受け入れる際に、変更が強制されることはない。また、必要に応じて追加のPreparerを作るのも簡単

### ダックタイピングの影響

最初の例：具象クラスに依存していた
具象性はコードを理解しやすくしてくれるが、<span style="background:#fff88f">拡張には危険が伴う</span>

直近の例：prepareはダックタイプに依存している
抽象的にした場合、理解力が求められるようになったが、<span style="background:#fff88f">拡張の容易さを提供してくれる</span>

<u>具象化のコストと抽象化のコストの間の緊張感</u>は、オブジェクト指向にとって基本的なこと

オブジェクトのクラスについての不明瞭さを多めに見るという能力は、自信を持った設計者であることを証明する
# 5.2ダックを信頼するコードを書く
設計上で難しいことは、ダックタイプが必要であることの気づくことと、そのインターフェースを抽象化すること
### 隠れたダックを認識する

以下の例はダックで置き換えることができる
- クラスで分岐するcase文
- kind_of?とis_a?
- responds_to?

#### クラスで分岐するcase文
```ruby
class Trip
  attr_reader :bicycles, :customers, :vehicle

  def prepare(preparers)
    preparers.each do |preparer|
      case preparer
      when Mechanic
        preparer.prepare_bicycles(bicycles)
      when TripCoordinator
        preparer.buy_food(customers)
      when Driver
        preparer.gas_up(vehicle)
        preparer.fill_water_tank(vehicle)
      end
    end
  end
end
```
アプリケーションが持つドメインオブジェクトのクラス名に基づいて分岐するcase文には深く注意を払う必要がある

ここでは、prepareメソッドは共通して、引数が旅行(trip)を準備することを望む

#### kind_of?とis_a?
この二つもcase分と同様にクラスを確認する方法である
```ruby
if preparer.kind_of?(Mechanic)
  preparer.prepare_bicycles(bicycle)
elsif preparer.kind_of?(TripCoordinator)
  preparer.buy_food(customers)
elsif preparer.kind_of?(Driver)
  preparer.gas_up(vehicle)
  preparer.fill_water_tank(vehicle)
end
```

#### responds_to?
```ruby
if preparer.respond_to?(:prepare_bicycles)
  preparer.prepare_bicycles(bicycle)
elsif preparer.respond_to?(:buy_food)
  preparer.buy_food(customers)
elsif preparer.respond_to?(:gas_up)
  preparer.gas_up(vehicle)
  preparer.fill_water_tank(vehicle)
end
```
このコードは依存の数は減らせているものの、未だ依存を持ちすぎている
クラス名こそなくなったが、<span style="background:#fff88f">コードは依然としてクラスに固く結びついている</span>

### ダックを信頼する
前述のダックで置き換えられる３つの例は、それぞれの場合で、コードは
「あなたは誰だか知っている、なぜならば、『あなたが何をするか知っているから』」と言っているのと同義である

デメテルの法則に違反するのと同じように、この形式のコードは<span style="background:#fff88f">何かのオブジェクトを見逃している</span>ことを示す。

柔軟なアプリケーションは信頼に基づいて働くオブジェクトによって作られる

ダックタイプを掴めたら、そのインターフェースを定義して、必要なところで実装をする。実装したら、それが<span style="background:#fff88f">正しく振る舞ってくれると信頼する</span>


#### ダックタイプを文書化する
最も単純なダックタイプ：パブリックインターフェースの取り決めとしてだけ存在するもの

ダックタイプを作る時は、そのパブリックインターフェースの文書化とテストを、両方しなければならない。<span style="background:#fff88f">優れたテストは最高の文書でもある</span>

### ダック間でコードを共有する
関数やメソッドの名前、引数の数やデータ型、返り値の型などの組み合わせのことをシグネチャ

一度ダックタイプを使い始めると、ダックタイプを実装するクラスは振る舞いもいくらか共有する必要があることに気づく（７章にて取り扱う）

### 賢くダックを選ぶ
kind_of? is_a?を使っても良い時がある
```ruby
# <tt>find(:first, *args)</tt> を便利に使うためのラッパー。
# このメソッドには <tt>find(:first)</tt> メソッドと
# 同じ引数をすべて渡せる
def first(*args)
  if args.any?
    if args.first.kind_of?(Integer) ||
       (loaded? && !args.first.kind_of?(Hash))
      to_a.first(*args)
    else
      apply_finder_options(args.first).first
    end
  else
    find_first
  end
end
```
今までの例と多く異なるのは、<span style="background:#fff88f">確認されているクラスの安定性</span>
Rubyのコアクラスへ依存をしている。この依存はItegerやHashが変わる可能性が極端に低いため、安全と言える

<font color="#ff0000">設計の目的は、コストを下げる事。</font>

基本クラスに変更を加えること：モンキーパッチ
これをするには、はっきりとその正当性を説明できなければいけない

# 5.3ダックタイピングへのおそれを克服する
### 静的型付けによるダックタイプの無効化
ほとんどの静的型付け言語は、それぞれの変数の型と、メソッドのすべてのパラメーターを明示的に宣言することを求める。動的型付け言語はこれを省く。Rubyは明らかに後者


型のエラーを気にしすぎた結果、動的型付け言語の良さを損なってしまうことになる。

この窮地から抜け出すのがダックタイピング。
<span style="background:#fff88f">クラスへの依存を取り除くことで、続発する方の失敗を避けることができる</span>。ダックタイプは、コードが安全に依存できる安定した抽象を明らかにする。

### 静的型付けと動的型付け

<span style="background:#b1ffff">静的型付け言語の特性</span>
- コンパイラがコンパイル時に型エラーを発見してくれる
- 可視化された型情報は、文書の役割を果たしてくれる
- コンパイルされたコードは最適化され、高速に動作する

以下の仮定がある時、上の三つは強みになる
- コンパイラが型を検査しない限り、実行時の型エラーが起きる
- 方がなければプログラマーはコードを理解できない。プログラマーはオブジェクトのコンテキストからその肩を推測することができない
- 一連の最適化がないとアプリケーションの動作は遅くなりすぎる

<span style="background:#b1ffff">動的型付け言語の特性</span>
- コードは逐次実行され、動的に読み込まれる。その為、コンパイル/makeのサイクルがない
- ソースコードは明示的な型情報を含まない
- メタプログラミングがより簡単

以下の過程がある時、上の三つは強みになる
- アプリケーション全体の開発は、コンパイル/makeのサイクルがない方が高速
- 型宣言がコードに含まれない時の方がプログラマーにとって理解がかんたん。そのコンテキストからオブジェクトの型は推測できる
- メタプログラミングは、あることが望ましい言語機能

### 動的型付けを受け入れる

疑いようもなく、一定のアプリケーションにおいては、動的型付け言語での実装よりも、上手に最適化された静的型付け言語のコードの方が優れた性能を発揮する。

Q：実装速度は誤字？

メタプログラミングは手術のメスのようなもの。正しく使える人が使うと強力な力を発揮する

静的型付け言語支持者の考え方
「予期しない型エラーを実行時に出されないことは、重要であり、かつ価値がある。コンパイラを取り除くことによって得られるプログラミングの大きな効率性を上回るほど、重要で価値のあることだ」
静的型付け言語支持者の考え方の前提
- コンパイラは不慮の型エラーから本当に救って「くれる」
- コンパイラの助けなしでは、これらの型エラーは「起こる」

動的型付け言語ではこうはならない。<span style="background:#fff88f">救ってくれないし、エラーは起きない</span>

型エラーを防ぐのはプログラマー自身。コードの良さは結局テストの良さになる。

動的型付けは、コンパイル時の型検査、つまりコストが高い割にわずかな価値しか得られない深刻な制約を、コンパイル/makeサイクルを取り除くことによって、得られる莫大な効率性と交換してくれる。しないてはない。
ダックタイピングを使うには、この動的さを受け入れる必要がある
