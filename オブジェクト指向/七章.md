# 7.1ロールを理解する

ロール（役割）：問題によってそれまで関連のなかったオブジェクト同士に共通の振る舞いを持たせなけらばならない時がある。この共通の振る舞いはクラスと<mark style="background:#affad1">直行する。</mark>

Q：直行とは、ここではどういうニュアンスか
A：

オブジェクトにロールを持たせると、オブジェクト間に依存関係が生じる

### ロールを見つける
5章で使ったダックタイプはロール

**モジュール**：クラスとは独立し、どんなオブジェクトにでも混ぜ入れることができるもの
モジュールはさまざまなクラスのオブジェクトが一箇所に定義されたコードを使った共通のロールを担うための完璧な方法と言える

オブジェクトが応答できるメッセージの集合
- 自信が実装するメッセージ
- 自信より上の階層の、すべてのオブジェクトで実装されるメッセージ
- 自信に追加されるすべてのモジュールで実装されるメッセージ
- 自信より上の階層に追加される、すべてのモジュールで実装されるメッセージ

### 責任を管理する

Scheduleクラスに以下の三つのメソッドがあるとする
```ruby
scheduled?(target,starting,ending)
add(target,starting,ending)
remove(target,starting,ending)
```
target =対象のオブジェクト
starting =開始日
ending =終了日

このコードだと、スケジュールできるかどうかを判断する情報としては不十分である。


これは、<mark style="background:#fff88f">Scheduleが自身の責任として正確なリードタイムを把握している場合の実装である</mark>＝知りすぎている

![[スクリーンショット 2026-03-01 16.50.23.png]]
他のクラスの詳細を知るのではなく、Scheduleは他のクラスにメッセージを送るべきである

### 不必要な依存関係を取り除く

![[スクリーンショット 2026-03-01 16.54.49.png]]クラス名の確認をScheduld?メソッドから取り除き、メソッドの代わりにlead_daysメッセージを送るように変化
これにより、正しいリードタイムの日数を把握する責任は、答えを一番知っているであろうオブジェクトに移された。

targetはいくつもあるクラスのどのクラスのインスタンスでもあり得る。なので、どのように名づければいいのかわからない

#### オブジェクト自信に自信を語らせる
新たに発見したダックタイプを使うことで、Scheduleの特定のクラス名への依存が除去され、コードは改善されるが、まだ不必要な依存がある。

<mark style="background:#fff88f">自身の振る舞いは自身が持つべきである</mark>
