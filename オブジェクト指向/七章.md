# 7.1ロールを理解する

ロール（役割）：問題によってそれまで関連のなかったオブジェクト同士に共通の振る舞いを持たせなけらばならない時がある。この共通の振る舞いはクラスと<mark style="background:#affad1">直行する。</mark>

Q：直行とは、ここではどういうニュアンスか
A：

オブジェクトにロールを持たせると、オブジェクト間に依存関係が生じる

### ロールを見つける
5章で使ったダックタイプはロール

**モジュール**：クラスとは独立し、どんなオブジェクトにでも混ぜ入れることができるもの
モジュールはさまざまなクラスのオブジェクトが一箇所に定義されたコードを使った共通のロールを担うための完璧な方法と言える

オブジェクトが応答できるメッセージの集合
- 自信が実装するメッセージ
- 自信より上の階層の、すべてのオブジェクトで実装されるメッセージ
- 自信に追加されるすべてのモジュールで実装されるメッセージ
- 自信より上の階層に追加される、すべてのモジュールで実装されるメッセージ

```ruby
class Schedule
  def scheduled?(schedulable, start_date, end_date)
    puts "This #{schedulable.class} " +
```

```ruby
class Bicycle
  attr_reader :schedule, :size, :chain, :tire_size

  # Scheduleを注入し、初期値を設定する
  def initialize(args = {})
    @schedule = args[:schedule] || Schedule.new
    # ...
  end

  # 与えられた期間の間、bicycleが利用可能であればtrueを返す
  def schedulable?(start_date, end_date)
    !scheduled?(start_date - lead_days, end_date)
  end

  # scheduleの答えを返す
  def scheduled?(start_date, end_date)
    schedule.scheduled?(self, start_date, end_date)
  end

  # bicycleがスケジュール可能となるまでの準備日数を返す
  def lead_days
    1
  end

  # ...
end

require 'date'
starting = Date.parse("2015/09/04")

b.schedulable?(starting, ending)
```

### 抽象を抽出する

<font color="#ff0000">コードを再構成し、クラスが異なっても振る舞いを共有できるようにする</font>

```ruby
module Schedulable
  attr_writer :schedule

  def schedule
    @schedule ||= ::Schedule.new
  end

  def schedulable?(start_date, end_date)
    !scheduled?(start_date - lead_days, end_date)
  end

  def scheduled?(start_date, end_date)
    schedule.scheduled?(self, start_date, end_date)
  end

  # 必要に応じてインクルードする側で置き換える
  def lead_days
    0
  end
end
```

Bicycleにあった時との変更点
- scheduleメソッドが追加された
- lead_day に初期値が設定された

これをすることで、Bicycleを以下のように変更することができる

```ruby
class Bicycle
  include Schedulable

  def lead_days
    1
  end

  # ...
end

require 'date'
starting = Date.parse("2015/09/04")
ending = Date.parse("2015/09/10")

b = Bicycle.new
b.schedulable?(starting, ending)
# This Bicycle is not scheduled
# between 2015-09-03 and 2015-09-10
# => true
```

![[スクリーンショット 2026-03-02 16.35.02.png]]

メッセージのパターンが変わった。
schedulable?をBicycleに送っていたものが、Schedulableに送るようになった


このような変化により、以下のように他のクラスにincludeすることで、Schedulableをどこでも使えるように！

```ruby
class Vehicle
  include Schedulable

  def lead_days
    3
  end

  # ...
end

class Mechanic
  include Schedulable

  def lead_days
    4
  end

  # ...
end

v = Vehicle.new
v.schedulable?(starting, ending)
# This Vehicle is not scheduled
# between 2015-09-01 and 2015-09-10
# => true

m = Mechanic.new
m.schedulable?(starting, ending)
# This Mechanic is not scheduled
# between 2015-02-29 and 2015-09-10
# => true
```


「である(is-a)」と「のように振る舞う(behaves-like-a)」の違いはかなり重要
どちらを使うかで、それぞれ及ぼす影響が異なる。しかし、実装するためのコーディング手法には類似度がある。この類似度の背景には、どちらも<mark style="background:#fff88f">自動的なメッセージの委譲に頼っているところにある</mark>

### メソッドの探索の仕組み

オブジェクトがメッセージを受け取ると、オブジェクト指向言語はまずオブジェクトの「クラス」を見に行き、合致するメソッドを探す

メソッドの探索は、メッセージを受け取ったオブジェクトのクラスから始まる。
そこになければ、探索はそのスーパークラスへと移る
<font color="#ff0000">探索はスーパークラスのチェーンを上へ上へと遡っていき、最終的に階層の頂点にたどり着くまで続く</font>。

最終的にObjectクラスを探索し、なかった場合、二つ目の手法にうつる

method_messageを送ることで、二回目のチャンスを与えている


####  より正確な説明

モジュールに定義されたメソッドは、メソッド探索パス上でBicycleに定義されたメソッド上に追加される

モジュールはインクルードされた時とは逆順で探索パスに配置される

### ロールの振る舞いを継承する

<mark style="background:#fff88f">下手に使ってしまうと、本当にひどいコードを作ってしまうこともできる</mark>

正しい使い方を学ぶための最初の一歩は、継承可能なコードを的確に書くこと



# 継承可能なコードを書く
以下の二つに出会った時、継承コードを使ってコードを改善できないか疑う
- オブジェクトがtypeやcategoryという変数名を使い、どんなメッセージをselfに送るか決めているパターン

新たな型を追加するたび、コードを変更しなくてはならない

- メッセージを受け取るオブジェクトのクラスを確認してから、どのメッセージを送るかをオブジェクトが決めているパターン

ダックタイプを見落としてしまう。受け取る側のオブジェクトを増やすたびに毎回コードを変更しなくてはならない

### 抽象に固執する

抽象スーパークラスないのコードを使わないサブクラスがあってはならない
すべてのサブクラスで使わないけれど、一部のサブクラスでは使うというようなコードはスーパークラスに置くべきでない

### 契約を守る
サブクラスは「契約」に同意する。スーパークラスと置換できることを約束する

SOLIDのL
「システムが正常であるためには、派生型は、上位型と置換可能でなければならない」


### テンプレートメソッドパターンを使う

テンプレートメソッドパターン：抽象的な処理と具体的な処理を分ける事で、変化に強い構造を実現

抽象コードでアルゴリズムを定義し、抽象を継承する、具象では、テンプレート化されたメソッドをオーバーライドする事で、特化を行う

### 前もって疎結合にする

継承する側でsuperを呼び出すようなコードを書くのは避ける
代わりにフックメッセージを使う

### 階層構造は浅くする

オブジェクトは、自信よりも上に位置するオブジェクト「全て」に依存するので、階層構造が深ければ、それだけ大量の依存を始めから持つことになる

階層構造が深いと、プログラマはその頂点と底辺のクラスばかりを理解しがちで、中間のクラスは軽くあしらわれる。
なんと泣くしか理解されていない中間層のクラスへの変更では、当然エラーが起きやすくなる
